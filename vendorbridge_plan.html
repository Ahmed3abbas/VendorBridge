Project Overview & Vision
Problem statement, core workflow, and system goals

VendorBridge is a Procurement & Vendor Management ERP that digitizes and centralizes the entire procurement lifecycle — from vendor onboarding through RFQ creation, quotation comparison, multi-level approval, purchase order generation, and GST-compliant invoice delivery. The platform replaces email-based procurement chaos with structured, auditable workflows.

Core Procurement Workflow
1
Procurement Officer creates an RFQ
Sets title, items + quantities, deadline, uploads attachments, assigns invited vendors
2
Vendors receive invitation & submit quotations
Each vendor submits line-item pricing, delivery timeline, and notes. Editable until deadline.
3
Officer compares quotations side-by-side
Comparison table highlights lowest price per item, delivery speed, and vendor rating
4
Selected quote enters approval workflow
Manager/Approver reviews context, adds remarks, approves or rejects with audit trail
5
Approval auto-generates Purchase Order
PO number auto-generated (PO-YYYY-NNNN), tax calculated, status tracked
6
Invoice generated from PO
GST line items, subtotal, tax, grand total. Download as PDF, print, or email directly.
7
Activity logged throughout
Every state change — RFQ creation, quote submission, approval, PO issue — logged with user, timestamp, and metadata

System Architecture
Request flow, module boundaries, and component interaction
Design principle: Each backend module is fully self-contained — its own routes, controller, and service file. This lets three people work in parallel with near-zero merge conflicts.
🌐 Request Lifecycle
1. Client
React SPA sends request via Axios instance with JWT in Authorization header
2. Express
Route hit → auth.middleware verifies JWT → role.middleware checks permissions
3. Validation
validate.middleware runs Zod schema against req.body — returns 400 on failure
4. Controller
Receives clean, validated data; calls service layer; does no DB logic itself
5. Service
All business logic here — Prisma queries, state transitions, email triggers
6. Activity Log
Every mutation auto-inserts into activity_logs with entity type + actor
7. Socket.io
On key events, server emits to relevant user rooms (approver notified on new approval)
8. Response
Standard envelope: { success, data, message, pagination }

Schema Overview
TablePrimary PurposeKey FieldsRelations
users Authentication and role management idPK, email, password_hash, role (ENUM), is_active, created_at → vendors (1:1 optional), rfqs, activity_logs
vendors Vendor registry with GST and contacts idPK, name, gst_number, category, status, rating, contact_email, contact_phone, address → rfq_vendors, quotations, purchase_orders
rfqs Request for Quotation records idPK, title, description, deadline, status (OPEN/CLOSED/CANCELLED), created_byFK → rfq_items, rfq_vendors, quotations
rfq_items Line items within an RFQ idPK, rfq_idFK, product_name, quantity, unit, description → quotation_items
rfq_vendors Junction: which vendors invited to which RFQ idPK, rfq_idFK, vendor_idFK, invited_at many-to-many junction
quotations Vendor response to an RFQ idPK, rfq_idFK, vendor_idFK, total_amount, delivery_date, notes, status (DRAFT/SUBMITTED/SELECTED/REJECTED) → quotation_items, approvals
quotation_items Line-level pricing per quotation idPK, quotation_idFK, rfq_item_idFK, unit_price, quantity, subtotal
approvals Approval workflow state per quotation idPK, quotation_idFK, approver_idFK, status (PENDING/APPROVED/REJECTED), remarks, acted_at → purchase_orders (on APPROVED)
purchase_orders Official PO generated on approval idPK, po_number (auto: PO-YYYY-NNNN), quotation_idFK, vendor_idFK, status, total_amount, tax_amount, tax_rate, issued_at → invoices
invoices Tax invoice generated from PO idPK, invoice_number (auto: INV-YYYY-NNNN), po_idFK, subtotal, tax_rate, tax_amount, total, status (DRAFT/SENT/PAID), sent_at
activity_logs Audit trail for all procurement events idPK, user_idFK, action (string), entity_type, entity_id, metadata (JSONB), ip_address, created_at polymorphic entity reference
attachments Files attached to RFQs or POs idPK, entity_type, entity_id, file_name, file_path, mime_type, file_size, uploaded_byFK polymorphic

8-Hour Sprint Timeline
Parallel workstreams with synchronization points and milestones
Critical path: Auth (M1, hour 1) → shared schemas (M1+M2, hour 1) → API modules (M1+M2, hour 2–5) → frontend integration (M3, hour 2–7). M3 can start building UI with mock data immediately; replaces mocks with real API calls as M1/M2 complete each module.
Hour 0 – 1  |  All hands: Foundation
Repository setup, environment, and scaffolding
M1: Create GitHub repo, set branch protection on main, Docker Compose running with PostgreSQL, Prisma init + complete schema (all 12 tables), first migration applied locally.
M2: Review and sign off Prisma schema, scaffold all 9 module folders with empty routes/controller/service files.
M3: Vite project created, TailwindCSS + shadcn installed, React Router configured with placeholder routes, sidebar and topbar shell built. Shared /schemas/ folder created.
Hour 1 – 3  |  Core API + Auth
Authentication complete, Vendor and RFQ APIs working
M1: Auth module complete (register, login, refresh, RBAC middleware). Vendor CRUD API + search/filter. RFQ creation + items + vendor invite.
M2: Quotation submission API. Approval state machine logic. Activity log auto-middleware written and attached to all routes.
M3: Login page with Zod validation. Dashboard page with static data. Vendor list page skeleton. Start RFQ creation form (multi-step).
Hour 3 – 5  |  Advanced Modules + UI Integration
First PR merges, real API calls in frontend
M1: Quotation comparison endpoint (returns side-by-side data). Socket.io rooms setup (one room per role + one per user). First PR: auth + vendor + RFQ modules merged to develop.
M2: PO auto-generation on approval. Puppeteer PDF for PO. Nodemailer setup (test with Mailtrap). Invoice CRUD API.
M3: Wire login + vendor pages to real API (React Query). Quotation comparison table (TanStack Table). Approval workflow UI with timeline. Notification bell connected to Socket.io.
Hour 5 – 6.5  |  Documents + Reports
PDF generation, email, and analytics complete
M2: Invoice PDF template (GST-compliant HTML). Email invoice endpoint. Reports API (spend trends, vendor performance aggregations). Second PR merged.
M3: PO and Invoice pages with download/email buttons. Reports page with Recharts (spend bar chart, vendor performance table). Activity log page.
Hour 6.5 – 7.5  |  Polish & Offline
UX polish, validation, responsive, Service Worker
M1: Review all APIs — consistent error messages, missing edge cases (deadline expired, duplicate quote).
M2: Test full workflow end-to-end. Fix any PDF rendering issues.
M3: Form validation error messages visible on all pages. Empty states added. Mobile responsiveness pass. Service Worker registered (cache dashboard + vendor list for offline view).
Hour 7.5 – 8  |  FREEZE + Demo Prep
Seed data, final commit, tag v1.0.0
Run seed script with demo users and a complete procurement workflow (RFQ → quotations → approval → PO → invoice). Do one full walkthrough as each role. Fix critical bugs only. Tag release. Update README with docker compose up quickstart.

Git & Version Control
Branch strategy, commit conventions, and PR rules
Branch Strategy
!
main
Production branch. Branch protection ON. No direct push. Only PR merges allowed. Minimum 1 reviewer approval required.
~
develop
Integration branch. All feature branches merge here. Test here before promoting to main. This is your safe staging ground.
+
feat/<member>/<feature-name>
e.g. feat/m1/auth-module, feat/m3/rfq-form, feat/m2/invoice-pdf. Created from develop, merged back to develop.
↺
fix/<issue-description>
Bug fix branches. Same review process as features. e.g. fix/pdf-rendering-blank-page
Conventional Commit Format
# Format: type(scope): short description feat(auth): add JWT refresh token rotation feat(rfq): implement vendor assignment on create feat(invoice): add Puppeteer PDF generation fix(quotation): correct GST rounding to 2 decimal places chore: add Husky pre-commit hooks docs: update API route table in README.md refactor(po): extract auto-number logic to utility test(auth): add Supertest for login endpoint style: run Prettier across client/src
PR Rules (non-negotiable)
  • ✓
    PR title follows conventional commit format — enforced by CI lint rule
  • ✓
    At least 1 reviewer (never the PR author) must approve before merge
  • ✓
    All ESLint errors must be resolved — no red squiggles get merged
  • ✓
    PR description includes: What changed, Why it was needed, How to test it locally
  • ✓
    Squash merge to main — keeps history clean and bisectable
  • ✓
    Delete feature branch immediately after merge
  • ✓
    Never merge your own PR — cross-review is mandatory, even under time pressure
